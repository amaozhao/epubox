# EPUBox：当前实现架构设计

## 1. 项目概述

EPUBox 是一个命令行 EPUB 翻译工具，目标是在尽量保留原始 HTML/XHTML 结构的前提下，使用大语言模型完成章节翻译、校对、回写与重新打包。

当前实现的重点不是“把所有 HTML 标签替换成占位符”，而是：

- 让 LLM 直接处理原始 HTML 片段
- 仅保护完全不需要翻译的代码/样式块
- 用 DOM + xpath 精确回写翻译结果
- 用 JSON checkpoint 支持断点续传和重跑

---

## 2. 当前主流程

```text
EPUB
  -> Parser.parse()
  -> PreCodeExtractor.extract()
  -> DomChunker.chunk()
  -> TranslatorWorkflow (translate -> proofread -> apply_corrections)
  -> DomReplacer.restore()
  -> Builder.build()
  -> 输出新的 EPUB
```

按职责展开后是：

1. `Parser` 解压 EPUB、读取 XHTML/HTML/XML/NCX 文件、规范化 HTML。
2. `PreCodeExtractor` 将 `<pre>` / `<code>` / `<style>` 作为原子块替换为 `[PRE:N]` / `[CODE:N]` / `[STYLE:N]`。
3. `DomChunker` 基于 DOM 收集可翻译块，并记录每个块对应的 xpath。
4. `TranslatorWorkflow` 执行翻译、校对和校对结果应用。
5. `DomReplacer` 在原始 DOM 上按 xpath 回写译文，再恢复代码/样式占位符。
6. `Builder` 修改语言与 CSS 字体设置，并重新打包为 EPUB。

---

## 3. 架构原则

### 3.1 只保护不可翻译内容

普通 HTML 标签不会被整体替换为占位符，LLM 直接看到诸如下面的原始结构：

```html
<p>Hello <em>World</em></p>
```

只有这类内容会被保护：

- `<pre>`
- `<code>`
- `<style>`

其原因是这些块通常不应该被翻译，而且一旦被模型改写，很容易破坏代码、样式或排版。

### 3.2 保护标签按原子块处理

`PreCodeExtractor` 的当前语义是“命中即整体替换，不再递归进入子树”。

例如：

```html
<pre><code>print("hi")</code></pre>
```

会变成：

```text
[PRE:0]
```

而不是同时产生 `[PRE:0]` 和内部的 `[CODE:0]`。

这条规则的意义是：

- 最终出现在 chunk 文本里的占位符，和真正需要后续恢复的占位符保持一致
- 避免嵌套保护标签被重复记账
- 降低校验和调试时的歧义

### 3.3 回写基于 DOM 定位，而不是字符串拼接

恢复阶段不是把所有 chunk 简单拼回去，而是：

- 解析原始 `item.content` 为 DOM
- 用 `chunk.xpaths` 找到原始元素
- 用翻译后的元素替换对应节点
- 最后恢复 `[PRE:N]` / `[CODE:N]` / `[STYLE:N]`

这样可以最大程度避免字符串拼接带来的标签错位和结构损坏。

## 4. 核心组件

### 4.1 `engine/epub/parser.py`

职责：

- 解压 EPUB
- 读取文档文件
- 用 BeautifulSoup 规范化 HTML/XML
- 运行 `PreCodeExtractor.extract()`
- 运行 `DomChunker.chunk()`
- 构建 `EpubBook` / `EpubItem` / `Chunk`
- 持久化 JSON checkpoint
- 检测并迁移旧版导航 chunk 为 `nav_text` 模式

当前主入口：`Parser.parse()`。

### 4.2 `engine/item/precode.py`

职责：

- 提取 `<pre>` / `<code>` / `<style>`
- 将它们替换为二级占位符
- 在最终回写完成后恢复这些原始块
- 提供占位符完整性验证和轻量容错修复能力

当前实现要点：

- `pre` / `code` / `style` 独立编号
- 命中后整体替换，不再递归进入其子树
- 恢复顺序为 `style -> code -> pre`
- 恢复时按索引从大到小替换，避免子串误匹配

### 4.3 `engine/item/chunker.py`

职责：

- 基于 DOM 生成可翻译块
- 为每个块记录 xpath
- 估算 token 数并按 `token_limit` 贪心合并
- 对导航文件采用 `nav_text` 特殊模式

当前实现有两种 chunk 模式：

- `html_fragment`：常规 HTML 片段模式
- `nav_text`：导航文本节点模式

`DomChunker` 的关键行为：

- 非导航文件：收集 `<body>` 下的块元素，并额外收集 `<head><title>`
- 导航文件：提取可翻译文本节点，生成 `[NAVTXT:N]` 标记来保证回写精确性
- 跳过纯占位符、纯图片等不可翻译节点
- `figure` 和 `nav` 被视为不可拆分容器

### 4.4 `engine/agents/workflow.py`

职责：

- 调用翻译模型
- 校验翻译结果结构
- 调用校对模型
- 应用校对建议
- 管理 chunk 状态流转
- 在需要时切换备用模型

当前三步工作流：

1. `translate_step`
2. `proofread_step`
3. `apply_corrections_step`

关键特性：

- 翻译阶段最多重试 3 次
- 内容安全错误会尝试 fallback 模型
- `html_fragment` 通过 `validate_translated_html()` 校验结构
- `nav_text` 通过导航标记一致性校验
- 会过滤掉会破坏 `[PRE:N]` / `[CODE:N]` / `[STYLE:N]` 的校对建议

### 4.5 `engine/epub/replacer.py`

职责：

- 将 chunk 的译文写回原始 DOM
- 恢复二级占位符
- 做最终 HTML/XML 结构校验
- 标记 `WRITEBACK_FAILED` 分块

当前写回策略：

- 常规 chunk：按 `chunk.xpaths` 对应回写
- 导航 chunk：按 `nav_targets` 中的 `xpath + text_index` 回写文本节点
- 回写完成后统一运行 `PreCodeExtractor.restore()`
- 最后运行 `verify_final_html()`，确保没有残留占位符且 XML 可解析

### 4.6 `engine/epub/builder.py`

职责：

- 更新 `.opf` 中的语言信息
- 修改 CSS 中的字体设置
- 重新打包输出 EPUB

当前行为：

- 更新 `dc:language` 和 `dcterms:language`
- 将正文类标签字体统一到 `STYuanti, serif`
- 将 `code/pre/math/...` 等标签字体设置为等宽字体

### 4.7 `engine/orchestrator.py`

职责：

- 串联解析、翻译、回写和打包流程
- 维护翻译统计
- 控制 checkpoint 保存
- 生成手动翻译报告
- 将输出写入 `extract_path + "_output"` 目录后再交给 `Builder`

## 5. 数据模型

### 5.1 `Chunk`

当前关键字段：

```python
class Chunk(BaseModel):
    name: str
    original: str
    translated: Optional[str]
    status: TranslationStatus
    tokens: int
    chunk_mode: Literal["html_fragment", "nav_text"]
    xpaths: List[str]
    nav_targets: List[NavTextTarget]
```

说明：

- `original` 是送给工作流的原始载荷
- `translated` 是翻译或校对后的结果
- `chunk_mode` 决定校验和回写逻辑
- `xpaths` 用于常规 HTML 片段回写
- `nav_targets` 用于导航文本节点回写

### 5.2 `EpubItem`

当前关键字段：

```python
class EpubItem(BaseModel):
    id: str
    path: str
    content: str
    translated: Optional[str]
    placeholder: Optional[Dict[str, str]]
    chunks: Optional[List[Chunk]]
    preserved_pre: Optional[List[str]]
    preserved_code: Optional[List[str]]
    preserved_style: Optional[List[str]]
```

说明：

- `content` 始终保留原始完整 HTML/XML
- `translated` 是最终恢复后的完整内容
- `preserved_pre` / `preserved_code` / `preserved_style` 用于二级占位符恢复
- `placeholder` 字段仍存在于 schema 中，但当前主流程已经不再依赖旧版全标签占位符体系

### 5.3 `TranslationStatus`

当前状态枚举：

```python
PENDING
TRANSLATED
ACCEPTED_AS_IS
COMPLETED
TRANSLATION_FAILED
WRITEBACK_FAILED
```

含义：

- `PENDING`：待翻译
- `TRANSLATED`：已翻译，待校对或待后续流程处理
- `ACCEPTED_AS_IS`：接受原文作为最终结果
- `COMPLETED`：翻译与校对完成
- `TRANSLATION_FAILED`：翻译失败，需要保留原文或人工处理
- `WRITEBACK_FAILED`：已有译文，但无法安全回写到最终 DOM

## 6. 占位符与校验策略

### 6.1 二级占位符格式

- `[PRE:N]`
- `[CODE:N]`
- `[STYLE:N]`
- `[NAVTXT:N]`（仅导航 chunk 内部使用）

### 6.2 占位符规则

- `PRE`、`CODE`、`STYLE` 各自独立编号
- 保护标签命中后按原子块整体替换
- 嵌套保护标签不会额外生成内部占位符
- 翻译和校对阶段必须原样保留这些占位符

### 6.3 结构校验

当前分两层：

1. **chunk 级校验**
   - HTML 标签数量和顶层元素结构是否一致
   - 占位符集合是否一致
   - 导航标记是否完整

2. **最终文件级校验**
   - 无残留 `[PRE:N]` / `[CODE:N]` / `[STYLE:N]`
   - 最终结果能通过 XML 解析

## 7. 分块与回写闭环

### 7.1 常规 HTML 文件

示例：

```html
<body>
  <h1>Title</h1>
  <p>Hello <em>world</em></p>
  <pre><code>print("hi")</code></pre>
</body>
```

经过 `PreCodeExtractor` 后：

```html
<body>
  <h1>Title</h1>
  <p>Hello <em>world</em></p>
  [PRE:0]
</body>
```

然后：

- `DomChunker` 会跳过纯占位符节点 `[PRE:0]`
- 仅把 `<h1>`、`<p>` 等可翻译元素打成 chunk
- 回写时只替换这些可翻译元素
- 最终再把 `[PRE:0]` 恢复为原始 `<pre><code>...</code></pre>`

### 7.2 导航文件

导航文件不直接按大块 HTML 翻译，而是提取文本节点：

```text
[NAVTXT:0] Chapter 1
[NAVTXT:1] Getting Started
```

回写时通过：

- 父元素 xpath
- 父元素内文本节点序号 `text_index`

来精准恢复到原始导航结构中。

## 8. 断点续传与失败处理

### 8.1 JSON checkpoint

`Parser.save_json()` 会把当前状态保存到与 EPUB 同名的 JSON 文件中。

保存内容包括：

- 书籍元数据
- 每个文档的原文与最终译文
- chunk 列表及其状态
- `preserved_pre` / `preserved_code` / `preserved_style`

### 8.2 重启恢复

`Parser.load_json()` 会优先读取 checkpoint。

此外还会做一件事：

- 如果检测到旧版导航 chunk，不直接沿用，而是重建为新的 `nav_text` 模式

### 8.3 手动翻译与失败分支

当 chunk 进入失败状态时：

- `TRANSLATION_FAILED`：说明模型未能给出合格译文
- `WRITEBACK_FAILED`：说明已有译文但无法安全写回 DOM

编排器会：

- 记录失败统计
- 允许部分状态在重跑时重新进入工作流
- 输出 `manual_translation_report.json` 供人工处理

## 9. 当前架构与旧架构的差异

### 已淘汰的核心思路

以下叙述不再代表当前主流程：

- “所有 HTML 标签都先替换成 `[idN]`”
- “通过 PlaceholderManager 管理全局标签占位符映射”
- “通过 TagPreserver / TagRestorer 做整章标签保护与恢复”
- “通过 Merger 先拼接 chunk 再统一恢复结构”
- “chunk 依赖全局索引 / 局部索引映射恢复标签”

### 当前真实实现

当前实现采用的是：

- 原始 HTML 直送 LLM
- 仅对 `pre/code/style` 做二级占位符保护
- DOM 分块
- xpath 回写
- 文件级最终校验

## 11. 总结

当前版本的 EPUBox 已从“全标签占位符驱动”切换为“DOM + xpath + 二级占位符”的实现。

最重要的几条事实是：

- LLM 直接处理原始 HTML，而不是纯文本加全标签占位符
- 只有 `pre/code/style` 会被保护
- 这些保护标签按原子块整体处理，不递归进入子树
- 分块依据是 DOM 结构与 token 限制
- 回写依据是 xpath，而不是字符串拼接
- 最终输出会经过占位符残留检查和 XML 结构校验

这份文档描述的是当前代码已经实现的主流程，不再保留旧架构作为默认实现说明。
